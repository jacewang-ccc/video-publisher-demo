# 项目决策摘要（防止聊天上下文满了丢失）

> 这份文件用于在对话“背景信息/上下文”接近 100% 时，快速恢复我们已确认的需求与方向。  
> 以后你只要把这份文件发给我（或让我读取它），我就能继续接着做，不用你重复描述。

## 1) 产品目标（MVP）
- 做一个“内容创建面板 + 自动化流程配置中心 + 多平台预览矩阵 + 扩展执行器”的工具。
- 你先在网页里确认所有平台最终要发布的内容，再“一键最终发布”（扩展去各平台准备与提交）。
- 当前阶段优先把“发布面板/配置中心/预览矩阵”的逻辑梳理清楚；扩展识别字段（不同平台简介位置不同）后续再做。

## 2) 发布流程（两阶段提交 2-phase publish）
- **Prepare（准备）**：锁定快照后，扩展后台打开各平台发布页并填充/上传，直到“可发布状态”。
- **Commit（最终发布）**：你在网页点一次“最终发布”，扩展统一点击发布；遇到扫码/验证码则暂停并提示你介入。
- 预览矩阵只展示“最终字段文本”（不做模拟手机预览）。

## 3) 内容分类结构（可扩展）
- 选择 **大主题 → 子主题**，且都支持新增。
- 预置：
  - 日更：表达/英语/唱歌/冥想/运动/阅读（可新增）
  - 副线：AI类/口才类/讲书（可新增）
  - 主线：碎碎念（可新增）

## 4) 自动化配置的核心（只用“自动化流程 pipeline”，不用 If/Then）
- 已决定：配置中心不再用 “If/Then 规则区”，所有自动化都用 **pipeline（流程）** 建模。
- pipeline 的维度是：**子主题 × 平台 × 功能(颗粒度)**。
  - 平台：抖音/小红书/视频号/B站（以及“全部”）
  - 功能：标题 / 简介 / tag（后续可扩展到：城市、发布时间、声明、合集等）

## 5) pipeline 的编辑方式（你要求的“拖拽式”先用列表版实现）
- 一个 pipeline 由 3–5（或更多）个“单元 step”组合而成，顺序可调整。
- 当前支持的 step（MVP）：
  - 固定文本（fixed）：例如 `Day`、`| `
  - 输入项引用（var）：例如 `打卡天数`（当前内部名 `day_n`）
  - 下拉选择（choice）：候选可编辑；候选只有 1 个时等同固定；可选“发布时手选/固定第1个/随机”
- 你希望的例子（表达×抖音×标题）：
  - `Day` + `{打卡天数}` + `| ` + 下拉(坚持表达/坚持练嘴/坚持表达能力提升/表达提升)

## 6) “反显”原则（非常重要）
- 配置中心配置出来的 pipeline **必须反显到创建内容页左侧**：
  - 哪些“输入项”需要你填（例如打卡天数）
  - 哪些“下拉项”需要你选
- 你必须先在创建内容页把这些填/选好，才能生成“多平台预览矩阵”的最终文本。

## 7) 优先级 priority（用来解决冲突）
- 同一“子主题+平台+功能”如果配置了多条 pipeline：
  - **优先级数字越大 → 越优先使用**
- 如果你保证永远只配一条，priority 可一直填 0。

## 8) 视觉与交互偏好（当前）
- 配置中心需要不同模块有颜色区分（已做淡色分区，避免全白晃眼）。
- 自动化配置交互顺序必须清晰：
  1) 先选 子主题/平台/功能/优先级
  2) 新增流程
  3) 在该流程内新增/删除单元、调顺序、改内容
  4) 保存后退出编辑状态

## 9) Extension 风险（先记下来，后做）
- 你担心扩展无法识别“简介/描述”填写位置，因为不同平台页面结构不同且会变动。
- 后续连接器建议策略：
  - 语义定位（找“作品描述/简介/正文”等文本附近控件）
  - 录制/修正（找不到时你点一次，记录路径）
  - 失败兜底（一键复制字段 + 高亮提示区域）
